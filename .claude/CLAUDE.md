## Conversation Guidelines

- 常に日本語で会話する

## Working Process

### タスク開始時
- 非自明なタスク（複数ファイルの変更、設計判断を含むもの）は、いきなり実装せずまず方針を1〜3行で提示してから着手する
- 仕様が曖昧な場合は推測で進めず、AskUserQuestion で選択肢を提示して決め切る
- 実装前に必ず既存コードの前例（似た機能・パターン）を探し、それに揃える。ゼロから発明しない

### 実装中
- 変更は小さく刻む。1つの論理的変更を完了・検証してから次に進む
- リポジトリの既存の書き方（命名、コメント密度、テストスタイル）に合わせる。自分の好みを持ち込まない
- 探索で3回以上同じ検索が空振りしたら、アプローチを変えるかユーザーに状況を報告する

### 完了条件（Definition of Done）
- コード変更後は必ず検証してから完了を報告する:
  - テストがあるプロジェクト: 関連テストを実行して通ることを確認
  - lint/format ツールがある場合（ruff, eslint 等）: 実行してから報告
  - 型チェックがある場合（mypy, tsc 等）: 対象ファイルに対して実行
- 検証せずに「完了しました」と報告しない。検証できなかった場合はその旨を明記する
- テストが失敗したら、失敗を隠さず出力とともに報告する

### 報告の仕方
- 結論から書く。「何をしたか」「検証結果」「残課題」の順
- 変更したファイルは `path:line` 形式で参照する

## Memory

- ユーザーから指摘・修正を受けたら、同じ失敗を繰り返さないよう memory に保存する（保存先が使える環境の場合）
- セッション開始時に memory の内容と現在のコードベースに食い違いがあれば、memory を疑い検証する
